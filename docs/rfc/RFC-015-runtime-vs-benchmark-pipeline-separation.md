RFC-015 Runtime vs Benchmark Pipeline Separation
================================================

Context and motivation
----------------------

The current benchmark stack intentionally prefers compact structured artifacts and algorithmic validation over full prose generation. That decision is correct for cost, latency, and determinism in benchmark runs.

The production runtime has a different goal: return a complete, user-facing answer. During the company-fact fast-path work, these two pipelines were partially mixed. As a result, the production answer for `Расскажи о компании` was rendered from a compact benchmark-oriented payload instead of from full-fidelity data.

The defect is now directly observed in the live stack:

- `corp_db_search` for `общая информация о компании ЛАДзавод светотехники` returns `result_format=compact_company_fact_v1`
- `core.tools.corp_db` truncates `preview` with `…`
- `core.agent` renders the final user answer from those truncated previews

That is why the user sees:

- `Мы занимаем одно из ведущих мест ... ЛАДзавод све…`
- `Наш профиль — промышленное освещение ... рассчи…`

The problem is not limited to one query. The architecture currently allows benchmark compaction to leak into any production path that renders from `ToolResult.output`.

Goals
-----

- Strictly separate runtime response payloads from benchmark artifacts.
- Keep benchmark validation cheap and deterministic.
- Preserve runtime optimizations in routing and retrieval without degrading the final answer.
- Ensure production final answers are generated from full-fidelity data.
- Keep deterministic fallback only as a resilience path, not as the default answer format for normal successful requests.

Non-goals
---------

- Redesign the benchmark framework.
- Remove algorithmic validation.
- Remove routing fast-paths or retrieval shortcuts.
- Change database schemas only to support this split.

Audit findings
--------------

The codebase audit finds one active production leak and several coupling points that make similar regressions likely.

Active production leak
~~~~~~~~~~~~~~~~~~~~~~

1. `core/tools/corp_db.py`

- `_compact_company_fact_payload()` truncates previews and drops full fields.
- `_format_result_payload()` returns that compact payload as `ToolResult.output` for company-fact `kb_search`.
- This is benchmark logic leaking into runtime output.

2. `core/agent.py`

- Company-fact fast-path and deterministic fallback render from `tool_result.output`.
- Because `tool_result.output` is already compacted, the final user answer is rendered from truncated previews.

Contract-coupling points
~~~~~~~~~~~~~~~~~~~~~~~~

3. `core/tests/test_corp_db_tool.py`

- Tests currently assert that company-fact runtime output is `compact_company_fact_v1`.
- This encodes the wrong contract and made the regression look intentional.

4. `bench/*`

- Benchmark runs correctly rely on `bench_artifact`, `primary_artifact`, `direct_tool`, and algorithmic checks.
- That is the right place for compact payloads.
- The risk is that tool authors may keep optimizing `ToolResult.output` for benchmark convenience because the benchmark and runtime share the same executor.

Already-correct reference pattern
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

5. `core/tools/doc_search.py`

- `ToolResult.output` contains the full payload.
- `metadata["bench_artifact"]` contains a separate compact artifact.
- This is the target pattern for all retrieval tools.

6. `core/run_meta.py` and `core/api.py`

- `bench_artifacts` are only attached when `return_meta=true`.
- This is debug/benchmark state, not user-facing state.
- The structure is acceptable; the issue is not in `run_meta`, but in reusing compact runtime output as if it were a bench artifact.

Design principles
-----------------

1. One tool call may produce multiple representations, but each representation has one owner and one purpose.
2. `ToolResult.output` is runtime-facing and full-fidelity by default.
3. `ToolResult.metadata["bench_artifact"]` is benchmark-facing and bounded.
4. Runtime optimization happens in routing, retrieval, and data selection, not by degrading the final answer payload.
5. The final user answer is generated by the LLM from full-fidelity evidence, except when the system explicitly enters a fallback mode.

Target architecture
-------------------

Runtime pipeline
~~~~~~~~~~~~~~~~

1. Classify intent.
2. Choose the minimal retrieval/tool chain.
3. Execute tools and collect full-fidelity structured results.
4. Pass those structured results to the final LLM answer step.
5. Return a complete user-facing answer.

Benchmark pipeline
~~~~~~~~~~~~~~~~~~

1. Execute `direct_tool`, `agent_chat_shadow`, or full `agent_chat`.
2. Collect compact bounded artifacts from `metadata["bench_artifact"]` or direct tool payload capture.
3. Validate algorithmically.
4. Ignore full prose unless the case explicitly requires prose evaluation.

Fallback pipeline
~~~~~~~~~~~~~~~~~

A deterministic renderer remains allowed only for:

- empty LLM completion
- transport failure
- explicit safe-mode exact-field fallback

It is not the default happy-path renderer for company facts, application подбор, document lookup, or portfolio answers.

Required contract changes
-------------------------

Tool contract
~~~~~~~~~~~~~

For retrieval tools such as `corp_db_search` and `doc_search`:

- `ToolResult.output` must contain full runtime payload.
- `ToolResult.metadata["bench_artifact"]` may contain compact bounded data for benchmark/debug.
- Optional `ToolResult.metadata["runtime_payload_kind"]` may describe the runtime format, but must not replace the runtime payload itself.

Agent contract
~~~~~~~~~~~~~~

- The normal production final answer step consumes full tool outputs.
- Fast-path routing may skip extra retrieval steps, but not the final LLM answer step.
- Deterministic rendering is allowed only in fallback mode.

Benchmark contract
~~~~~~~~~~~~~~~~~~

- Bench evaluators consume `bench_artifact`, `primary_artifact`, or direct tool capture.
- Bench must never require production `ToolResult.output` to be compact for algorithmic validation to work.

Implementation outline
----------------------

Phase 1: Stop the active leak
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

1. Change `core/tools/corp_db.py` so company-fact searches return full JSON payload in `ToolResult.output`.
2. Keep compact company-fact payload only inside `metadata["bench_artifact"]`.
3. Update `core/tests/test_corp_db_tool.py` to assert the new contract.

Phase 2: Separate runtime rendering from fallback rendering
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

4. Change `core/agent.py` so normal company-fact answers go through the final LLM step with full evidence.
5. Keep deterministic fallback for `empty_llm_completion`, transport failures, and similar degraded states.
6. Add an explicit runtime flag in routing state such as `finalizer_mode=llm|deterministic_fallback`.

Phase 3: Generalize the split
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

7. Audit all retrieval tools under `core/tools/` and ban benchmark-specific compaction in `ToolResult.output`.
8. Use `core/tools/doc_search.py` as the reference implementation pattern.
9. Add a small helper or policy check so new tools cannot accidentally reuse benchmark compactors for runtime output.

Phase 4: Protect with tests and benchmark coverage
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

10. Add unit tests that fail if production output contains benchmark-only fields such as `result_format=compact_*`.
11. Add end-to-end live cases for:
    - `Расскажи о компании`
    - `Подскажи контакты компании.`
    - one application подбор
    - one document lookup
12. Add a regression check that runtime answers do not contain truncation ellipses introduced by tool compactors unless the source text itself already contains them.

Detailed code impact
--------------------

Files to change in the first implementation round:

- `core/tools/corp_db.py`
- `core/agent.py`
- `core/tests/test_corp_db_tool.py`
- `core/tests/test_routing_guardrail.py`
- optionally `bench/bench_run.py` or `bench/bench_lib.py` only if a benchmark assumption still depends on compact runtime output

Files already aligned and to be used as examples:

- `core/tools/doc_search.py`
- `core/run_meta.py`

Operational and observability notes
-----------------------------------

The separation should be visible in telemetry:

- runtime logs should show whether the final answer path is `llm` or `deterministic_fallback`
- tool spans should record `runtime_payload_format=full` and `bench_artifact_present=true|false`
- production incidents should be diagnosable without enabling benchmark-specific response formats

Testing approach
----------------

Unit tests
~~~~~~~~~~

- `corp_db_search` returns full runtime payload for company-fact searches.
- `bench_artifact` stays compact and bounded.
- agent company-fact path uses full data in normal mode.
- deterministic fallback still works when LLM is empty.

Integration tests
~~~~~~~~~~~~~~~~~

- `/api/chat` for `Расскажи о компании` returns complete text without benchmark truncation.
- `/api/chat` for `Подскажи контакты компании.` still returns exact fields.
- benchmark `direct_tool` and `algorithmic` runs remain green after the contract split.

Manual checks
~~~~~~~~~~~~~

- Telegram voice/text query for `Расскажи о компании`
- Telegram query for `Контакты`
- one stadium/application подбор
- one certificates/document lookup

Acceptance criteria
-------------------

1. No production retrieval tool uses benchmark compaction to shape `ToolResult.output`.
2. `corp_db_search` company-fact runtime output is full-fidelity JSON.
3. Benchmark still receives compact bounded artifacts through metadata.
4. Normal production company-fact answers are finalized by the LLM from full evidence.
5. Deterministic rendering happens only in fallback mode.
6. `Расскажи о компании` no longer returns internally truncated previews with `…` introduced by benchmark compactors.
7. Existing benchmark pass rate does not regress after the split.
